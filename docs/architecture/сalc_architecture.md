# Расчет архитектуры

Давай проведем расчет архитектуры.

**Расчёт RPS (запросов в секунду) и пикового RPS.**

RPS будем рассчитывать по формуле:
$$
  RPS = (DAU * Q) / t
$$
где:

```text
DAU - количество уникальных пользователей в сутках;
Q - среднее число запросов на пользователя в сутки (по ролям: Q_сотр, Q_инж);
t - количество секунд в сутках = 86400.
```

В соответствии требованиям к системе US-01, US-02 определено следующие количество уникальных пользователей DAU: сотрудников ~1000, плюс до 50 инженеров.
Давайте создадим сводную таблицу типов запросов с конкретной привязкой их к ролям, для более точного расчета среднего числа запросов в сутки.

## Сводная таблица типов запросов для расчёта RPS

| ID | Роль |Тип запроса |Основной сценарий | Ориентировочная частота |
|----|------|------------|------------------|--------------------------|
| A1 | Сотрудник, Инженер | Вход | Начало сессии | По числу входов в единицу времени |
| A2 | Сотрудник, Инженер | Выход | Завершение сессии | По числу выходов |
| A3 | Сотрудник, Инженер | Сессия/профиль | Проверка при загрузке/обновлении страницы | Высокая при активных сессиях |
| B1 | Сотрудник, Инженер | Список теплиц | Выбор теплицы (сотрудники, инженеры) | При открытии/смене контекста |
| B2 | Сотрудник, Инженер | Камеры теплицы | Выбор камеры | При выборе теплицы |
| B3 | Сотрудник, Инженер | Live-поток | Просмотр видео в реальном времени | 1 на активный просмотр |
| B4 | Сотрудник, Инженер | Показания датчиков (текущие) | Обновление блока датчиков (≤30 с) | Периодически при открытом просмотре |
| C1 | Инженер | Поиск событий | Запрос списка событий по критерию | По действию «Поиск» (инженеры) |
| C2 | Инженер | Детали события | Открытие карточки события | При выборе из списка |
| C3 | Инженер | Архивное видео | Просмотр архива по событию | При открытии архива |
| C4 | Инженер | Динамика датчиков | График по событию | При открытии архива |
| D1 | Сотрудник, Инженер | Регионы/локации | Выбор по иерархии | При загрузке формы выбора |

Сотрудники (~1000) генерируют в основном запросы A1–A3, B1–B4, D1; инженеры (~50) — дополнительно C1–C4.

Итого получаем (суммарно 8600 запросов в сутки):
$$
  RPS = \frac{N_{\text{сотр}} \cdot Q_{\text{сотр}} + N_{\text{инж}} \cdot Q_{\text{инж}}}{t} = \frac{8 \cdot 1050 + 4 \cdot 50}{86400} = \frac{8600}{86400} \approx 0{,}1.
$$

Введем пиковый коэффициент K = 2 (учёт смен и кратковременных всплесков нагрузки). Итого пиковый RPS:

$$
  RPS_{\text{pik}} = RPS \cdot K \approx 0{,}2.
$$

**Расчёт CCU (количество одновременных соединений) и пикового CCU.**

Расчёт CCU будем вести по формуле:

$$
CCU = (DAU * s) / tm;
$$
где:

```text
N_сотр - количество сотрудников в сутках (US-01, NFR-04, ~1000);
N_инж - количество инженеров в сутках (US-02, NFR-04, ~50);
s_сотр - средняя длительность сессии сотрудника (мин);
s_инж - средняя длительность сессии инженера (мин);
tm - количество минут в сутках = 1440.
```

Уточним допущения по длительности сессий (на основе сценариев US-01 и US-02):

- сотрудник (US-01) обычно заходит в интерфейс для онлайн‑мониторинга теплицы на **15 минут**;
- инженеру (US-02) требуется больше времени на анализ данных и просмотр архивного видео — возьмём **30 минут**.

Тогда расчёт CCU по ролям:

$$
  CCU = \frac{N_{\text{сотр}} \cdot s_{\text{сотр}} + N_{\text{инж}} \cdot s_{\text{инж}}}{tm} = \frac{1000 \cdot 15 + 50 \cdot 30}{1440} = \frac{16500}{1440} \approx 11{,}5.
$$

Введём тот же пиковый коэффициент K = 2. Пиковый CCU:

$$
  CCU_{\text{pik}} = CCU \cdot K \approx 23.
$$

## Виды соединений и оценка одновременных подключений

С учётом рассчитанных RPS и CCU и требований NFR-01…NFR-04 можно оценить основные типы соединений и их одновременное количество.

1. **HTTP(S)/WebSocket (браузер ↔ веб/API):**
   - Среднее CCU ≈ **12** одновременных пользовательских сессий.
   - Пиковое CCU ≈ **23** сессии.
   - Для каждой сессии предполагается не более 1–2 активных вкладок/стримов на пользователя.

2. **MQTT (контроллеры теплиц ↔ брокер):**
   - Около **1000 теплиц** (BR-R01, BR-R02), по одному контроллеру на теплицу.
   - Оценка: ≈ **1000 постоянных TCP‑соединений** к MQTT‑брокеру.

3. **RTSP/медиа‑потоки (камеры ↔ медиа‑сервер):**
   - До **5 цифровых камер** на теплицу (BR-R02), то есть теоретически до 5000 RTSP‑потоков.
   - Фактически одновременно активными будут потоки только по просматриваемым теплицам/камерам; при 20–25 одновременных пользователях и 1–2 потоках на пользователя — **до 50 одновременных клиентских видеопотоков**.

4. **Соединения с БД и хранилищем:**
   - Пулы соединений с СУБД на каждом инстансе веб/API‑слоя (десятки соединений на инстанс).
   - Доступ к хранилищу видео и архивных данных датчиков — через внутренние сервисы/драйверы (нагрузка зависит от реализации, но существенно ниже по количеству одновременных клиентских подключений, чем MQTT/RTSP).

## Итоги по нагрузке и необходимым серверам

С учётом рассчитанных нагрузок (RPS ≈ 0,1, RPS_{\text{pik}} ≈ 0,2; CCU ≈ 12, CCU_{\text{pik}} ≈ 23) и масштаба (NFR-04: 1000+ теплиц, ~1050 пользователей) можно предложить следующую минимально достаточную конфигурацию серверов (без привязки к конкретному вендору или облаку):

- **Веб-/API‑слой:**
  - Не менее **2 инстансов веб/API‑приложения** за балансировщиком нагрузки для отказоустойчивости и горизонтального масштабирования по RPS.

- **Медиа-/стриминговый слой (RTSP → клиенты):**
  - **1–2 медиа‑сервера**, принимающих RTSP от камер и раздающих потоки пользователям (live и архив), с возможностью горизонтального масштабирования по числу одновременных видеопотоков.

- **MQTT‑слой:**
  - **Кластер из 2 брокеров MQTT** (основной + резервный / HA‑конфигурация), рассчитанный примерно на 1000 постоянных соединений от контроллеров теплиц.

- **Слой хранения данных:**
  - СУБД: **1 основной сервер БД + 1 реплика** для отказоустойчивости и разгрузки чтения (поиск событий, отчёты).
  - Хранилище для видео и архивных данных датчиков (объектное/файловое) — масштабируется по объёму, не является узким местом по количеству одновременных соединений.

Такая конфигурация покрывает текущие оценочные нагрузки и требования NFR-01 (задержка до 30 с), NFR-02 (отклик поиска до 10 с), NFR-03 (доступность 99,9%) и NFR-04 (масштабируемость на 1000+ теплиц и ~1050 пользователей) с запасом для последующего горизонтального масштабирования.
