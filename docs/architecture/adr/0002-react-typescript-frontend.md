## Контекст

### Проблема

Системе нужны два браузерных SPA: основное веб-приложение ИС (`CNT_GM_Web`) и клиент подсистемы Identity (`CNT_GM_Identity_Web`). Пользователи просматривают live-видео и показания датчиков, выполняют поиск событий и работают с архивом, проходят аутентификацию через OpenID Connect. Интерфейсы опираются на **HTTPS**, **WebSocket**, **GraphQL** и потоковое видео в браузере (в т.ч. WebSocket/MSE к медиа-сервисам по модели [cnt_gm_web](../diagram/containers/cnt_gm_web/01-model.c4)).

Нужно зафиксировать стек реализации клиентской части, чтобы единообразно развивать UI, типизировать контракты с backend и не плодить несовместимые фреймворки.

### Предпосылки

- В LikeC4 для `CNT_GM_Web` заданы **React**, **TypeScript** и целевая версия **React 18.2.0** (метаданные модели).
- `CNT_GM_Identity_Web` описан как React/TypeScript без отдельной версии — принимается тот же стек, что и у основного SPA.
- Серверная часть зафиксирована в [ADR-0001](0001-dotnet-aspnet-core-backend.md); клиент должен предсказуемо интегрироваться с OIDC и API.

### Ограничения

- Технические: работа в современных браузерах, ограничения по задержке отображения данных и видео ([NFR-01](../../requirements/non-functional/01-latency-live-stream.md)).
- Организационные: один основной фронтенд-стек на продукт, чтобы не размывать код-ревью, дизайн-систему и CI для фронта.

## Рассмотренные альтернативы

### Вариант A: React и TypeScript (принятый)

**Описание:** SPA на React с компонентной моделью; вся прикладная логика UI на TypeScript для статической типизации и контрактов с API/GraphQL.

**Плюсы:**

- Зрелая экосистема компонентов, инструментов сборки и отладки.
- TypeScript снижает риск регрессий при изменении DTO, GraphQL-схем и маршрутов.
- React 18 даёт современные возможности рендеринга и распространённый на рынке навык.

**Минусы:**

- Размер бандла и сложность конфигурации сборки выше, чем у минимального HTML/JS.
- Обновления major-версии React требуют регрессии по ключевым сценариям (live, поиск, архив).

**Оценка:**

```yaml
cost: Низкий
complexity: Средняя
time_to_market: Средний
risks: ["Миграции major React", "Дисциплина типов при росте кодовой базы"]
```

### Вариант B: Другой фреймворк (Vue, Angular, Svelte)

**Описание:** Реализовать SPA на альтернативном фреймворке с TypeScript.

**Плюсы:**

- Возможный выбор под привычки части команды или встроенные решения (например, структура Angular).

**Минусы:**

- Расхождение с уже принятыми контейнерами C4 и [tech-stack.md](../../ai/tech-stack.md).
- Дополнительные затраты на переопределение архитектурных артефактов и обучение.

**Оценка:**

```yaml
cost: Средний
complexity: Средняя
time_to_market: Выше за счёт смены направления
risks: ["Несогласованность с репозиторием и документацией"]
```

### Вариант C: React без TypeScript (JavaScript)

**Описание:** Тот же React, но без обязательной типизации.

**Плюсы:**

- Меньше дисциплины типов на старте.

**Минусы:**

- Слабее гарантии на границе с GraphQL и REST при росте числа экранов и сценариев.
- Не соответствует заявленной в архитектуре связке React + TypeScript.

**Оценка:**

```yaml
cost: Низкий
complexity: Низкая на старте, растёт со временем
time_to_market: Быстрый старт
risks: ["Ошибки контрактов с backend", "сложнее рефакторинг"]
```

---

## Решение

Принять **React** и **TypeScript** как **единый** стек для браузерных приложений в границах системы: **`CNT_GM_Web`** и **`CNT_GM_Identity_Web`**.

Целевая версия React для основного приложения — **18.2.0** (как в метаданных [cnt_gm_web](../diagram/containers/cnt_gm_web/01-model.c4)); для Identity Web использовать ту же мажорную линию React, что и у `CNT_GM_Web`, если иное не оформлено отдельным решением.

Новые клиентские SPA в составе этой системы вводятся на **React + TypeScript**, если иное не оформлено отдельным ADR.

## Последствия

### Положительные

- Единый подход к компонентам, состоянию и тестированию UI для мониторинга и Identity.
- Типобезопасность на стыке с API и упрощение сопровождения при эволюции GraphQL/WebSocket.
- Согласованность с [ADR-0001](0001-dotnet-aspnet-core-backend.md): клиент потребляет те же контракты, что описаны на сервере.

### Отрицательные

- Необходимость отслеживать обновления React и экосистемы (сборщик, линтер, типы).
- Фронтенд-разработка остаётся на стеке, отличном от .NET — две экспертизы в команде (ожидаемо для web-приложений).

### Последующие действия

- Держать [tech-stack.md](../../ai/tech-stack.md) и метаданные LikeC4 (`version` в `cnt_gm_web`) согласованными с этим ADR.
- При смене мажорной версии React или отказе от TypeScript оформить новый ADR или пересмотр статуса этого документа.
