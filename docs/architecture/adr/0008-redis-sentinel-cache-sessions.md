## Контекст

### Проблема

Слой **`CNT_GM_WebAPI`** проектируется как **stateless** при горизонтальном масштабировании (**не менее двух инстансов** за балансировщиком): пользовательское состояние сессий должно храниться **вне** процесса API ([01-calc-architecture.md](../01-calc-architecture.md)). Параллельно **`CNT_GM_SavingService`** обновляет **кэш последних показаний датчиков** для быстрого чтения онлайн-показаний в **`CNT_GM_Redis_DB`** ([01-model.c4](../diagram/containers/cnt_gm_redis_db/01-model.c4)).

Нужно зафиксировать:

1. использование **Redis** как контейнера кэша/сессий;
2. схему **высокой доступности** при отказе узла Redis — без обязательности sticky sessions на балансировщике для обычного HTTP.

### Предпосылки

- В LikeC4 для **`CNT_GM_Redis_DB`** указаны **Redis** и роли: запись кэша из **SavingService**, чтение и сессии из **WebAPI** ([01-model.c4](../diagram/containers/cnt_gm_redis_db/01-model.c4)).
- В расчёте архитектуры явно указано: сессии — в **Redis**; sticky sessions **не обязательны** при такой модели ([01-calc-architecture.md](../01-calc-architecture.md)).
- Целевое развёртывание: Redis и Sentinel как поды в **одном Kubernetes-кластере** с приложениями ([30-production-deployment.c4](../diagram/infrastructure/30-production-deployment.c4)).

### Ограничения

- Redis **не** является системой записи «истины» по телеметрии: долговременное хранение — **ClickHouse** ([ADR-0004](0004-clickhouse-telemetry.md)); Redis — кэш и сессии с политикой TTL/перезаполняемости.
- Конфигурация клиентов .NET должна указывать на **текущий мастер** после failover (через Sentinel или сервисный слой Kubernetes — детали в техническом проекте).

## Рассмотренные альтернативы

### Вариант A: Redis (primary/replica) + Redis Sentinel (принятый)

**Описание:** Один логический контур **Redis**: экземпляр **primary**, **реплика** для чтения и резерва данных в памяти; **несколько процессов Redis Sentinel** (рекомендуется **нечётное** число, на диаграмме — **три**) для мониторинга мастера, **автоматического failover** и публикации актуального адреса мастера клиентам.

**Плюсы:**

- Привычная модель HA для Redis без перехода на **Redis Cluster** там, где не требуется шардирование по ключам.
- Соответствует цели **NFR-03**: отказ одного узла Redis не должен делать недоступным сессии и кэш при корректной настройке Sentinel и клиентов.
- Совместимо с развёртыванием в **Kubernetes** (отдельные StatefulSet/Deployment для Redis, реплики и Sentinel).

**Минусы:**

- Дополнительные процессы **Sentinel** и их согласованная конфигурация (кворум, `quorum`, сетевые политики).
- Необходимость тестирования сценариев failover для **WebAPI** и **SavingService**.

**Оценка:**

```yaml
cost: Низкий–средний
complexity: Средняя
time_to_market: Средний
risks: ["Split-brain при сетевых сбоях без правил размещения", "таймауты клиентов при переключении мастера"]
```

### Вариант B: Один экземпляр Redis без Sentinel

**Описание:** Упрощённая установка Redis в одном поде или VM.

**Плюсы:**

- Минимум компонентов для dev/test.

**Минусы:**

- **Единая точка отказа** для сессий и кэша — неприемлемо для целевого **production** при требованиях к доступности ([NFR-03](../../requirements/non-functional/03-availability.md)).

**Оценка:**

```yaml
cost: Низкий
complexity: Низкая
time_to_market: Быстрый
risks: ["Простой Redis = потеря сессий и кэша при сбое узла"]
```

### Вариант C: Управляемый облачный Redis (managed cache)

**Описание:** Сервис вида AWS ElastiCache, Azure Cache for Redis и т.п.

**Плюсы:**

- Меньше операционной нагрузки на команду (патчи, резервное копирование по SLA провайдера).

**Минусы:**

- Зависимость от облака и стоимости; текущая диаграмма инфраструктуры ориентирована на **самостоятельный** кластер Kubernetes с in-cluster Redis.

**Оценка:**

```yaml
cost: Средний–высокий (OPEX)
complexity: Низкая–средняя
time_to_market: Средний
risks: ["Vendor lock-in", "расхождение с выбранной моделью деплоя в K8s"]
```

### Вариант D: Redis Cluster

**Описание:** Шардирование данных по нескольким мастерам.

**Плюсы:**

- Горизонтальное масштабирование по объёму ключей и пропускной способности.

**Минусы:**

- Избыточная сложность при проектной нагрузке из [01-calc-architecture.md](../01-calc-architecture.md) (RPS/CCU умеренные; основной рост данных — в ClickHouse, не в Redis).

**Оценка:**

```yaml
cost: Средний
complexity: Высокая
time_to_market: Выше
risks: ["Операции resharding", "сложнее клиентская конфигурация"]
```

## Решение

Принять **Redis** как реализацию контейнера **`CNT_GM_Redis_DB`**: **кэш последних показаний датчиков** (запись **`CNT_GM_SavingService`**, чтение **`CNT_GM_WebAPI`**) и **хранилище пользовательских сессий** для stateless-инстансов API.

Для **production** зафиксировать связку **репликации Redis (primary + replica)** и **Redis Sentinel** в **нечётном количестве** (на целевой диаграмме — **три Sentinel**) для мониторинга и **автоматического выбора мастера** при отказе.

Развёртывание — **в Kubernetes** в том же кластере, что и прикладные сервисы ([30-production-deployment.c4](../diagram/infrastructure/30-production-deployment.c4)); изоляция по **namespace** и сетевые политики — по среде.

Клиенты (**ASP.NET Core**, см. [ADR-0001](0001-dotnet-aspnet-core-backend.md)) должны использовать конфигурацию, совместимую с **обнаружением мастера** после failover (Sentinel-aware клиент или эквивалент в инфраструктуре).

## Последствия

### Положительные

- Согласованность с LikeC4 и расчётом архитектуры (сессии в Redis, несколько инстансов WebAPI без обязательных sticky sessions).
- Понятная модель HA без Redis Cluster до появления реальной потребности в шардировании кэша.

### Отрицательные

- Дополнительные компоненты (**Sentinel**, реплика) и runbook на failover, резервное копирование (если политика требует сохранять сессии между катастрофами — уточнять приоритет: сессии часто допускается потерять при major incident).
- Необходимость нагрузочных и хаос-тестов на переключение мастера.

### Последующие действия

- Уточнить в техническом проекте: политики **TTL** для ключей кэша датчиков, шифрование трафика (**TLS** там, где уже зафиксировано в модели связей WebAPI–Redis), лимиты памяти и **eviction**.
- Держать [tech-stack.md](../../ai/tech-stack.md) и порты/версии Redis согласованными с эксплуатацией.
- При смене стратегии (например, только managed cache или Redis Cluster) оформить новый ADR и обновить [30-production-deployment.c4](../diagram/infrastructure/30-production-deployment.c4).
