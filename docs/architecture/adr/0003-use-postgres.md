## Контекст

### Проблема

Прикладной слою (`CNT_GM_WebAPI`) нужно надёжное **реляционное** хранилище для **справочников и метаданных**: теплицы, камеры, датчики, пользовательские и административные сущности, связи между ними. Данные должны храниться **без ограничения по сроку**, с **ACID-транзакциями** при изменении связей и справочников. Объёмы по [расчёту архитектуры](../01-calc-architecture.md) — порядка **десятков МБ – единиц ГБ**, нагрузка по соединениям — **десятки** активных подключений из пулов приложений; для отказоустойчивости рассматривается схема **1 основной + 1 реплика** ([NFR-03](../../requirements/non-functional/03-availability.md)).

Нужно явно зафиксировать выбор СУБД для контейнера **`CNT_GM_DB`**, отделив её роль от колоночного хранилища телеметрии (ClickHouse) и от объектного хранилища видео.

### Предпосылки

- В LikeC4 для `CNT_GM_DB` указаны **PostgreSQL** и описание: «База данных для хранения справочников и метаданных».
- В том же контуре **Identity** использует отдельную БД **`CNT_GM_Identity_DB`** на **PostgreSQL** — единый класс СУБД упрощает эксплуатацию (резервное копирование, мониторинг, патчи), хотя базы логически разделены.
- Backend на **.NET** ([ADR-0001](0001-dotnet-aspnet-core-backend.md)) имеет зрелую поддержку PostgreSQL через стандартные драйверы и ORM.

### Ограничения

- Не смешивать в одной физической базе прикладные метаданные и потоковую телеметрию — временные ряды остаются в **ClickHouse** (`CNT_GM_Timeseries_DB`).
- Сохранять возможность **репликации** и резервного копирования без привязки к проприетарному облаку.

## Рассмотренные альтернативы

### Вариант A: PostgreSQL (принятый)

**Описание:** Основная СУБД для `CNT_GM_DB` — **PostgreSQL**.

**Плюсы:**

- Полноценная реляционная модель, ограничения целостности, транзакции для справочников и связей.
- Зрелая репликация и экосистема для HA (в духе расчёта «1+1» в [01-calc-architecture.md](../01-calc-architecture.md)).
- Широкая поддержка в .NET и в инфраструктуре (Kubernetes, облака, on-prem).
- Совпадение с **`CNT_GM_Identity_DB`** снижает число различных СУБД в эксплуатации.

**Минусы:**

- Требуется администрирование (версии, миграции, индексы, vacuum/анализ по нагрузке).

**Оценка:**

```yaml
cost: Низкий
complexity: Средняя
time_to_market: Средний
risks: ["Плановые обновления major PostgreSQL"]
```

### Вариант B: MySQL / MariaDB

**Описание:** Реляционная альтернатива с похожей моделью использования.

**Плюсы:**

- Распространённость, хорошая поддержка в ORM.

**Минусы:**

- Расхождение с уже выбранным стеком **PostgreSQL** для Identity и в диаграмме `CNT_GM_DB` без выигрыша для объёма метаданных в этом проекте.

**Оценка:**

```yaml
cost: Низкий
complexity: Средняя
time_to_market: Средний
risks: ["Две семейства СУБД вместо одного", "несогласованность с LikeC4"]
```

### Вариант C: Microsoft SQL Server

**Описание:** Реляционная СУБД в экосистеме Microsoft.

**Плюсы:**

- Тесная интеграция с Windows-ландшафтом (если бы он был обязателен).

**Минусы:**

- Лицензирование и привязка к стеку отличаются от типичного Linux-контейнерного развёртывания PostgreSQL.
- Не отражено в текущих моделях контейнеров.

**Оценка:**

```yaml
cost: Средний или выше
complexity: Средняя
time_to_market: Средний
risks: ["Лицензии", "расхождение с архитектурной документацией"]
```

---

## Решение

Принять **PostgreSQL** как СУБД для **`CNT_GM_DB`**: хранение **справочников и метаданных** приложения (CRUD через `CNT_GM_WebAPI` по модели связей).

Использование **PostgreSQL** для **`CNT_GM_Identity_DB`** остаётся согласованным решением по классу СУБД (отдельные инстансы/базы, не смешивать данные Identity и прикладные метаданные без явного обоснования).

Телеметрия датчиков, поиск событий и аналитика по временным рядам **не** размещаются в `CNT_GM_DB` — для этого используется **ClickHouse** (`CNT_GM_Timeseries_DB`).

## Последствия

### Положительные

- Одна семья СУБД для прикладных метаданных и Identity на уровне технологии.
- Прямое соответствие [cnt_gm_db/01-model.c4](../diagram/containers/cnt_gm_db/01-model.c4) и расчётам в [01-calc-architecture.md](../01-calc-architecture.md).
- Предсказуемая модель масштабирования чтения через реплику при росте нагрузки на отчёты (в рамках NFR-03).

### Отрицательные

- Необходимость миграций схемы и регрессии при обновлениях PostgreSQL.
- Отдельное проектирование индексов и пулов соединений на инстансах Web API.

### Последующие действия

- Синхронизировать строки про PostgreSQL в [tech-stack.md](../../ai/tech-stack.md) и в [project-context.md](../../ai/project-context.md) при изменении границ данных.
- При смене СУБД для `CNT_GM_DB` оформить новый ADR и обновить LikeC4-модель `cnt_gm_db`.
