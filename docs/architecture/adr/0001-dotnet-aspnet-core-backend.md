## Контекст

### Проблема

Системе мониторинга теплиц нужна серверная платформа для нескольких связанных сервисов: основной Web API (HTTPS, WebSocket, GraphQL), фоновый приём и сохранение телеметрии из брокера, подсистема Identity на базе OpenIddict. Сервисы должны оставаться **stateless** за балансировщиком, масштабироваться горизонтально и единообразно сопровождаться ([NFR-03](../../requirements/non-functional/03-availability.md), [NFR-04](../../requirements/non-functional/04-scalability.md)).

Нужно зафиксировать выбор основной платформы реализации backend, чтобы не распылять стек и не умножать операционные риски.

### Предпосылки

- В архитектуре уже заложены **ASP.NET Core** и **.NET 10** для `CNT_GM_WebAPI`, `CNT_GM_SavingService` и `CNT_GM_Identity_WebAPI`.
- Identity строится на **OpenIddict** — зрелая интеграция с ASP.NET Core.
- Команда ожидает единый набор практик: конфигурация, логирование, health checks, деплой контейнеров.

### Ограничения

- Технические: необходимость стабильной работы с PostgreSQL, ClickHouse, Redis, RabbitMQ, Vault по документированным протоколам.
- Организационные: ограничение числа различных рантаймов и стеков на уровне операций и обучения команды.

## Рассмотренные альтернативы

### Вариант A: .NET / ASP.NET Core (принятый)

**Описание:** Один major-версии .NET для всех перечисленных сервисов; веб-слой на ASP.NET Core; OpenIddict для OIDC/OAuth2.

**Плюсы:**

- Один рантайм и экосистема для API, ingestion и Identity.
- Встроенная поддержка DI, конфигурации, middleware, интеграции с OpenIddict без «склейки» разных фреймворков.
- Предсказуемая модель горизонтального масштабирования stateless-приложений в Kubernetes или аналогах.

**Минусы:**

- Привязка к дорожной карте Microsoft / .NET; миграция major-версий требует планирования.
- Для части специалистов на рынке сравнима с Java, но не универсальна как «только Node» в frontend-командах.

**Оценка:**

```yaml
cost: Низкий
complexity: Средняя
time_to_market: Средний
risks: ["Зависимость от цикла релизов .NET LTS"]
```

### Вариант B: Несколько стеков (например, Node.js для API + отдельный сервис на другом языке)

**Описание:** Разделить сервисы по разным рантаймам по мере появления предпочтений команд.

**Плюсы:**

- Гибкий выбор библиотек под узкие задачи.

**Минусы:**

- Дублирование инфраструктуры наблюдаемости, секретов, паттернов деплоя.
- Выше стоимость владения и риск регрессий на границах стеков.

**Оценка:**

```yaml
cost: Высокий
complexity: Высокая
time_to_market: Выше за счёт разнообразия
risks: ["Фрагментация экспертизы", "Сложнее единые стандарты безопасности"]
```

### Вариант C: Java / Kotlin (Spring и т.п.)

**Описание:** Единая JVM-платформа для backend-сервисов.

**Плюсы:**

- Зрелая экосистема для enterprise и интеграций.

**Минусы:**

- Не совпадает с текущей моделью Identity на OpenIddict/ASP.NET без дополнительной переработки архитектуры.
- Дополнительный контур знаний, если команда уже ориентируется на .NET.

**Оценка:**

```yaml
cost: Средний
complexity: Средняя
time_to_market: Средний
risks: ["Расхождение с уже принятыми контейнерами C4 и tech-stack"]
```

---

## Решение

Принять **.NET (текущая целевая версия в архитектуре — .NET 10)** и **ASP.NET Core** как **единственную** платформу для серверных приложений Greenhouse Monitoring в границах системы: `CNT_GM_WebAPI`, `CNT_GM_SavingService`, `CNT_GM_Identity_WebAPI`.

Новые backend-сервисы в рамках этой системы добавляются на том же стеке, если иное не оформлено отдельным ADR.

---

## Последствия

### Положительные

- Единые паттерны безопасности, конфигурации и наблюдаемости.
- Упрощённая интеграция Identity с основным API (общие подходы к токенам, middleware).
- Меньше вариантов при разборе инцидентов и при онбординге разработчиков.

### Отрицательные

- Обновления major-версии .NET требуют регрессионного тестирования всего backend-контура.
- При найме ориентир на .NET-разработчиков для серверной части (не обязательно минус при планировании).

### Последующие действия

- Держать [tech-stack.md](../../ai/tech-stack.md) и LikeC4-модели согласованными с этим ADR.
- При смене целевой версии .NET оформить новый ADR или обновление статуса этого документа.
