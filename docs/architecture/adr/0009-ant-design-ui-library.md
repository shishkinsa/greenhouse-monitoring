## Контекст

### Проблема

После выбора **React** и **TypeScript** ([ADR-0002](0002-react-typescript-frontend.md)) для `CNT_GM_Web` нужно зафиксировать **библиотеку готовых UI-компонентов** и подход к **темизации**, чтобы:

- быстро собирать административные и операторские экраны (таблицы, формы, навигация, уведомления);
- обеспечить **единообразие** интерфейса и снизить объём самописного CSS;
- опереться на проверенные паттерны **доступности** и **локализации** (в т.ч. русский интерфейс);
- не плодить несколько несовместимых дизайн-систем в одном SPA.

Без явного решения риск — смесь headless-библиотек, самописных виджетов и несогласованных стилей, что усложняет ревью и онбординг.

### Предпосылки

- В репозитории уже используется **Ant Design** в проекте [cnt_gm_web](../../../src/frontend/cnt_gm_web) (`antd`, `@ant-design/icons`, `ConfigProvider`, `theme`, русская локаль).
- Типичные сценарии ИС мониторинга — **плотные таблицы**, фильтры, формы настроек, дашборды; Ant Design изначально ориентирован на такие **enterprise**-интерфейсы.
- Лицензия **MIT** у текущей линии `antd` снимает блокеры по коммерческому использованию при стандартной due diligence.

### Ограничения

- Размер **бандла** выше, чем у минимального набора headless-компонентов; требуется дисциплина импортов и при необходимости **ленивая загрузка** тяжёлых экранов.
- Визуальный язык Ant Design узнаваем и **менее гибок** «с нуля», чем полностью кастомная дизайн-система на токенах без библиотеки.
- Обновления **major**-версий `antd` требуют регрессии UI и проверки breaking changes в теме и API компонентов.

## Рассмотренные альтернативы

### Вариант A: Ant Design (принятый)

**Описание:** Полноценная React-библиотека компонентов с встроенной темой (Design Token), формами, таблицами, навигацией и локалями.

**Плюсы:**

- Богатый набор **Table**, **Form**, **Layout**, **DatePicker** и др. из коробки для админ-панелей.
- **ConfigProvider** + кастомизация токенов — единая точка темы без разрозненных констант.
- Официальная **локализация** (в т.ч. `ru_RU`) и документация на английском/китайском с активным сообществом.
- Согласуется с уже начатой реализацией `cnt_gm_web`.

**Минусы:**

- Крупный runtime и стили; без code splitting отдельные экраны раздувают первый чанк.
- Кастомный «брендовый» визуал требует работы поверх токенов и иногда обходных стилей.

**Оценка:**

```yaml
cost: Низкий
complexity: Низкая для стандартных экранов
time_to_market: Низкий
risks: ["Major-обновления antd", "размер бандла при росте числа страниц"]
```

### Вариант B: MUI (Material UI)

**Описание:** Популярная React-библиотека в духе Material Design.

**Плюсы:**

- Зрелая экосистема, гибкая тема, хорошая документация.

**Минусы:**

- Смена уже принятого в коде направления (Ant Design) без явной выгоды для текущего продукта.
- Другой визуальный язык и API — дополнительные затёки на миграцию.

**Оценка:**

```yaml
cost: Средний
complexity: Средняя
time_to_market: Средний
risks: ["Миграция существующего UI", "дублирование компетенций если часть экранов останется на Ant"]
```

### Вариант C: Headless (Radix UI / React Aria) + Tailwind CSS

**Описание:** Примитивы доступности и утилитарные стили; визуал и композиция полностью на команде.

**Плюсы:**

- Максимальный контроль над внешним видом и размером бандла по части CSS.

**Минусы:**

- Значительно больше времени на **таблицы**, сложные формы и единую дизайн-систему с нуля.
- Выше риск расхождения реализации с ожиданиями по срокам MVP мониторинга.

**Оценка:**

```yaml
cost: Средний
complexity: Высокая
time_to_market: Высокий
risks: ["Недостаток готовых enterprise-компонентов", "поддержка собственной дизайн-системы"]
```

### Вариант D: Mantine / Chakra UI

**Описание:** Современные альтернативные full-stack UI-kit’ы на React.

**Плюсы:**

- Удобный DX, хорошая типизация, активное развитие.

**Минусы:**

- Те же затраты на смену стека, что и у MUI, при уже выбранном Ant Design в коде.

**Оценка:**

```yaml
cost: Средний
complexity: Средняя
time_to_market: Средний
risks: ["Миграция", "несогласованность с документацией спеки frontend"]
```

## Решение

Принять **Ant Design** (`antd`) как **стандартную** библиотеку компонентов для SPA **`CNT_GM_Web`** (`cnt_gm_web`) в связке с **React** и **TypeScript** ([ADR-0002](0002-react-typescript-frontend.md)).

- Тема и глобальные настройки Ant — через **`ConfigProvider`** и централизованный объект темы (например, [shared/config/theme.ts](../../../src/frontend/cnt_gm_web/src/shared/config/theme.ts)); при необходимости обёртка **`App`** из `antd` в провайдерах слоя `app` для статического контекста `message` / `notification`.
- Локаль интерфейса Ant — согласованно с требованиями продукта (по умолчанию поддерживается сценарий с **русской** локалью там, где это задано в приложении).
- Для **`CNT_GM_Identity_Web`** при отсутствии отдельного ADR допускается тот же выбор (**Ant Design**) ради единообразия операторских интерфейсов; при выборе иной библиотеки оформить отдельное архитектурное решение.

Новые экраны в `cnt_gm_web` по умолчанию реализуются на **компонентах Ant Design** и токенах темы, а не на параллельной полной дизайн-системе без обоснования.

## Последствия

### Положительные

- Ускорение разработки типовых **CRUD**, таблиц телеметрии, форм настроек и навигации.
- Предсказуемые **паттерны** для команды и ревьюеров; согласованность со [спекой frontend](../specs/frontend/12-frontend-app-architecture.md).
- Возможность опираться на готовые **a11y**-практики компонентов при корректном использовании API.

### Отрицательные

- Зависимость от релизного цикла и breaking changes **antd**; нужен регресс при обновлении major.
- Ограничения по «уникальному» визуалу без инвестиций в кастомизацию токенов и обёртки.
- Размер бандла — учитывать в **Vite**/**code splitting** для крупных страниц.

### Последующие действия

- Держать зафиксированную мажорную линию `antd` согласованной с [package.json](../../../src/frontend/cnt_gm_web/package.json) и при необходимости обновить [tech-stack.md](../../ai/tech-stack.md).
- При смене библиотеки UI или отказе от Ant Design в пользу другой системы оформить **новый ADR** и обновить статус этого документа на «Заменено».
