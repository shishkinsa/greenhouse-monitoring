## Контекст

### Проблема

Несколько компонентов системы нуждаются в **секретах** (ключи API, пароли к внешним системам, учётные данные для камер и т.п.), которые **нельзя** хранить в исходном коде и нежелательно размножать в статичных конфигурационных файлах на узлах без центральной политики.

В архитектуре выделен отдельный контейнер **`CNT_GM_Secrets`**, а связи задают доступ по **HTTPS** и **Vault API** или с **Bearer token** к API Vault (см. модели ниже). Нужно зафиксировать продукт для этого контейнера и принцип: **единая точка** выдачи и учёта секретов для сервисов.

### Предпосылки

- В LikeC4 для `CNT_GM_Secrets` указаны **HashiCorp Vault** и описание «Контейнер для хранения секретов» ([model.c4](../diagram/containers/cnt_gm_secrets/model.c4)).
- **`CNT_GM_WebAPI`** обращается за секретами по **HTTPS, Vault API** ([model.c4](../diagram/containers/cnt_gm_webapi/model.c4)).
- **`CNT_GM_WebRTCServer_Only`** и **`CNT_GM_WebRTCServer_History`** получают секреты из Vault (в подписях связей — HTTPS / Bearer access token к Vault API по [моделям](../diagram/containers/cnt_gm_webrtcserver_only/model.c4)).

### Ограничения

- Секреты не должны подменять **OIDC-токены пользователей** для бизнес-API — это отдельный контур (Identity); Vault закрывает **секреты приложений и инфраструктуры**.
- Решение должно допускать **аудит** обращений и **ротацию** по политике безопасности (детали — в эксплуатационной документации Vault).

## Рассмотренные альтернативы

### Вариант A: HashiCorp Vault (принятый)

**Описание:** **HashiCorp Vault** как централизованное хранилище секретов с **HTTP API**, политиками доступа и поддержкой нескольких **backend**-движков (KV, PKI и др. по мере необходимости).

**Плюсы:**

- Явно зафиксировано в диаграмме и [tech-stack.md](../../ai/tech-stack.md).
- Единый контур для **`WebAPI`** и **go2rtc**-сервисов без собственных «сейфов» в каждом репозитории.
- Зрелая модель **политик**, **аудита** и интеграций с оркестраторами и CI/CD (по выбору команды).

**Минусы:**

- Операционная сложность: **HA**, **распечатка (unseal)**, резервное копирование и обновления кластера Vault.
- Требуется дисциплина: не дублировать секреты в переменных окружения в открытом виде при наличии Vault.

**Оценка:**

```yaml
cost: Низкий–средний
complexity: Средняя–высокая
time_to_market: Средний
risks: ["Доступность Vault = зависимость сервисов при старте", "корректная настройка TTL и ротации"]
```

### Вариант B: Только секреты облака (AWS Secrets Manager, Azure Key Vault)

**Описание:** Хранить секреты в управляемом сервисе выбранного облака без Vault.

**Плюсы:**

- Меньше администрирования кластера секретов при использовании managed-режима.

**Минусы:**

- Не совпадает с текущей фиксацией **HashiCorp Vault** в `CNT_GM_Secrets` без смены ADR и диаграммы.
- При **гибриде** или смене облака — несколько разных API секретов или дополнительная абстракция.

**Оценка:**

```yaml
cost: Зависит от облака и объёма API-вызовов
complexity: Средняя
time_to_market: Средний
risks: ["Vendor lock-in", "расхождение с LikeC4"]
```

### Вариант C: Kubernetes Secrets / переменные окружения без центрального хранилища

**Описание:** Секреты только в Secret-ресурсах k8s или в env на ВМ.

**Плюсы:**

- Простота на малых средах.

**Минусы:**

- Нет единого **Vault API**, описанного в моделях; слабее централизованный аудит и политики между кластерами и вне k8s (медиа-сервисы).
- Расхождение с архитектурной связью **HTTPS, Vault API** для Web API.

**Оценка:**

```yaml
cost: Низкий
complexity: Низкая на старте
time_to_market: Быстрый
risks: ["Фрагментация секретов", "сложнее единые политики ротации"]
```

---

## Решение

Принять **HashiCorp Vault** как реализацию контейнера **`CNT_GM_Secrets`**: централизованное хранилище секретов с доступом сервисов по **Vault API** (HTTPS), в соответствии с LikeC4.

**`CNT_GM_WebAPI`**, **`CNT_GM_WebRTCServer_Only`** и **`CNT_GM_WebRTCServer_History`** получают секреты только через этот контур (конкретные пути KV/engine и метод аутентификации приложений — в техническом проекте и секретной документации, не в публичной архитектуре).

## Последствия

### Положительные

- Одна согласованная точка для секретов приложений, совместимая с [cnt_gm_secrets/model.c4](../diagram/containers/cnt_gm_secrets/model.c4).
- Возможность расширения (PKI, динамические учётные записи БД и т.д.) без смены базового продукта.

### Отрицательные

- Зависимость доступности и производительности Vault от сценариев старта сервисов и кэширования секретов на стороне клиентов.
- Необходимость обучения команды модели политик Vault и процедур incident response.

### Последующие действия

- Держать [tech-stack.md](../../ai/tech-stack.md) согласованным с этим ADR.
- При замене Vault на другой продукт оформить новый ADR и обновить модель `cnt_gm_secrets` и все связи потребителей.
