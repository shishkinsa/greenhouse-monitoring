## Контекст

### Проблема

Архивное видео с камер теплиц записывается **сегментами** и должно храниться в **объектном** хранилище с **S3 API**: **`CNT_GM_WebRTCServer_History`** читает и пишет сегменты по **HTTPS, S3 API** ([01-model.c4](../diagram/containers/cnt_gm_webrtcserver_history/01-model.c4)). Объёмы по [01-calc-architecture.md](../01-calc-architecture.md) — **до ~162 ТБ за 30 дней** (верхняя оценка; фактически меньше при VBR, записи по событиям, снижении FPS и lifecycle). Для таких объёмов в расчёте явно выделено **S3-совместимое** хранилище: масштабирование по ёмкости, **низкая стоимость за ГБ**, **lifecycle** и классы хранения ([NFR-04](../../requirements/non-functional/04-scalability.md)).

Нужно зафиксировать продукт для контейнера **`CNT_GM_S3`**, сохраняя **контракт S3 API** для медиа-слоя и не привязывая архитектуру к одному публичному облаку без альтернативы on-prem.

### Предпосылки

- В LikeC4 для `CNT_GM_S3` указаны **MinIO** и описание «S3 хранилище для архива видео с камер» ([01-model.c4](../diagram/containers/cnt_gm_s3/01-model.c4)).
- Интеграция идёт через стандартный **S3 API** (не проприетарный SDK одного вендора на стороне протокола).

### Ограничения

- Не хранить крупные видеосегменты в PostgreSQL или ClickHouse — только метаданные и пути при необходимости в реляционной/аналитической БД.
- Сохранить возможность смены backend при сохранении **S3 API** (совместимость клиентов go2rtc и приложений).

## Рассмотренные альтернативы

### Вариант A: MinIO (принятый)

**Описание:** **MinIO** как реализация **S3-совместимого** объектного хранилища для `CNT_GM_S3` (развёртывание on-prem, в частном облаке или у совместимых провайдеров — по среде).

**Плюсы:**

- Нативный **S3 API**, ожидаемый связью `WebRTCServer_History` → `CNT_GM_S3`.
- Подходит под крупные объёмы и горизонтальное масштабирование ёмкости в духе расчёта V_video.
- Возможность **self-hosted** без обязательной привязки к одному публичному гиперскейлеру; политики **lifecycle** и классы хранения настраиваются в операционной модели MinIO/совместимых кластеров.

**Минусы:**

- Эксплуатация кластера MinIO (обновления, эрasure coding, мониторинг, резервное копирование — по политике проекта).
- Для публичного облака часто используют нативный **Amazon S3**; **MinIO** в ADR отражает **целевую** реализацию в модели контейнеров; при развёртывании в AWS/Azure допускается **S3-совместимый** endpoint при сохранении того же контракта API — оформляется в техническом проекте без смены диаграммы «логического» `CNT_GM_S3`.

**Оценка:**

```yaml
cost: Низкий–средний
complexity: Средняя
time_to_market: Средний
risks: ["Ёмкость и сеть при росте до оценок V_video", "тестирование совместимости S3-клиента go2rtc"]
```

### Вариант B: Только управляемый Amazon S3 (или аналог без self-hosted)

**Описание:** Отказ от MinIO в пользу только облачного S3.

**Плюсы:**

- Меньше операционной нагрузки на кластер объектного хранилища при использовании managed-сервиса.

**Минусы:**

- Не совпадает с явной технологией **`MinIO`** в текущей модели `CNT_GM_S3` без изменения ADR и диаграммы.
- Зависимость от региона/облака и модели ценообразования — отдельное решение по среде.

**Оценка:**

```yaml
cost: Зависит от облака и объёма
complexity: Низкая со стороны ops при managed
time_to_market: Быстрый при готовом аккаунте
risks: ["Vendor lock-in", "расхождение с LikeC4 при фиксации MinIO"]
```

### Вариант C: Файловое NAS вместо S3

**Описание:** Хранить сегменты на общей ФС без S3 API.

**Минусы:**

- Связь в архитектуре задана как **HTTPS, S3 API**; отказ потребовал бы смены интеграции медиа-сервера и модели.

**Оценка:**

```yaml
cost: Средний
complexity: Высокая
time_to_market: Высокий
risks: ["Полная переработка ingestion архива"]
```

---

## Решение

Принять **MinIO** как целевую технологию для контейнера **`CNT_GM_S3`**: **S3-совместимое** объектное хранилище для **архива видео** с камер, доступное по **S3 API** для **`CNT_GM_WebRTCServer_History`** (запись сегментов и чтение при воспроизведении).

Логически контейнер остаётся **объектным хранилищем с S3 API**; физическое размещение (выделенный кластер MinIO, совместимый сервис в облаке с тем же API) определяется средой развёртывания и не меняет контракт для приложений, пока сохраняется **S3 API**.

## Последствия

### Положительные

- Соответствие [cnt_gm_s3/01-model.c4](../diagram/containers/cnt_gm_s3/01-model.c4) и выводам [01-calc-architecture.md](../01-calc-architecture.md) про объектное S3-хранилище для видеоархива.
- Переносимость клиентов, использующих S3 API, между совместимыми backend при необходимости.

### Отрицательные

- Отдельный контур эксплуатации объектного хранилища (квоты, политики жизненного цикла, мониторинг места и исходящего трафика при воспроизведении архива).

### Последующие действия

- Держать [tech-stack.md](../../ai/tech-stack.md) и строки про MinIO / S3 согласованными с этим ADR.
- При смене продукта для `CNT_GM_S3` или отказе от S3 API оформить новый ADR и обновить LikeC4-модель `cnt_gm_s3`.
