# Модель угроз: система мониторинга теплицы

**Версия:** 1.0  
**Статус:** черновик архитектуры (подлежит пересмотру при изменении границ системы и ADR)

## Назначение и аудитория

Документ фиксирует **ожидаемые угрозы** и **направления защиты** для информационной системы мониторинга теплицы (далее — ИС). Он предназначен для архитекторов, разработчиков бэкенда и фронтенда, DevOps/SRE и ответственных за информационную безопасность.

Цель — не исчерпывающий penetration test, а **согласованная основа** для проектирования контролей, приоритизации доработок и трассировки требований к безопасности.

**Связанные артефакты:**

- Контекст и контейнеры: `docs/architecture/diagram/context/`, `docs/architecture/diagram/containers/`
- Поток аутентификации: [oidc-user-authentication.c4](./oidc-user-authentication.c4)
- Технологический стек: [tech-stack.md](../../../ai/tech-stack.md)
- ADR по платформе, данным, брокеру, объектному хранилищу, секретам, кэшу: `docs/architecture/adr/0001`–`0008`

## Область и допущения

| Элемент | Описание |
|--------|----------|
| **Входит в область** | Веб-клиенты (`CNT_GM_Web`, `CNT_GM_Identity_Web`), API (`CNT_GM_WebAPI`, `CNT_GM_Identity_WebAPI`), ingestion и сохранение данных (`CNT_GM_SavingService`), брокер MQTT/AMQP, PostgreSQL (метаданные и Identity), ClickHouse (телеметрия), Redis (кэш и сессии), MinIO (архив видео), HashiCorp Vault, go2rtc и связанные видеопотоки, reverse proxy / балансировщик по моделям деплоя |
| **Частично / по среде** | Физическая безопасность теплицы, сети на объекте, прошивки контроллеров и камер — учитываются как **внешние доверенные/недоверенные зоны**, детали зависят от эксплуатации |
| **Вне области документа** | Конкретные SLA провайдера, сертификация помещений, политики HR; они могут ссылаться на выводы этого документа |

Допущение по модели угроз: **нарушитель** может быть внешним (Интернет), инсайдером с ограниченными правами, скомпрометированным устройством на периметре теплицы или злоумышленником с доступом к резервным копиям/логам, если политики хранения слабы.

## Активы и последствия компрометации

| Актив | Примеры | Целостность / конфиденциальность / доступность |
|-------|---------|--------------------------------------------------|
| **Учётные данные и токены** | Пароли, refresh/access JWT, сессии в Redis, клиенты OIDC | Утечка ведёт к несанкционированному доступу к API и данным; подмена — к действиям от имени пользователя |
| **Метаданные и справочники** | PostgreSQL (`CNT_GM_DB`): объекты, привязки датчиков, конфигурации | Искажение — ложные показания и решения; утечка — раскрытие топологии и персональных/производственных данных |
| **Телеметрия** | ClickHouse: временные ряды датчиков | Подмена/удаление — срыв мониторинга и реакций; агрегированная утечка — коммерческая разведка |
| **Видеоархив и потоки** | MinIO, go2rtc, камеры | Нарушение конфиденциальности персонала/процессов; отказ — слепые зоны безопасности и контроля |
| **Секреты приложений** | Vault: строки подключения, ключи API | Полная компроментация зависимых сервисов |
| **Сообщения IoT** | MQTT/AMQP, очереди | Подмена телеметрии, DoS брокера, анализ трафика конкурентами |
| **Аудит и логи** | Журналы приложений, Vault, СУБД | Удаление следов после атаки; утечка — чувствительные данные в логах |

## Акторы и доверительные границы

**Акторы (из контекстной модели и эксплуатации):**

- **Сотрудник / инженер** — легитимные пользователи SPA, OIDC Authorization Code + PKCE ([oidc-user-authentication.c4](./oidc-user-authentication.c4)).
- **Администратор платформы** — доступ к инфраструктуре, Vault, CI/CD (модель прав не детализирована в ADR; трактуется как высокий привилегированный контур).
- **Устройства теплицы** — цифровые контроллеры по MQTT (TLS), камеры по RTSP/WebRTC — **периметр объекта**, частично недоверенный.
- **Внешний нарушитель** — сеть Интернет до reverse proxy и публичные точки входа.

**Границы доверия (упрощённо):**

1. Браузер пользователя ↔ reverse proxy: TLS, ограничение методов и заголовков.
2. SPA ↔ Identity Web API: только по HTTPS; токены не должны утекать в сторонние скрипты (политика CSP и хранение — зона ответственности фронтенда).
3. Web API ↔ Identity: валидация JWT (JWKS, issuer, audience).
4. Внутренняя сеть сервисов ↔ PostgreSQL, ClickHouse, Redis, RabbitMQ, MinIO, Vault: **сегментация**, TLS где предусмотрено архитектурой, не размещать БД в публичной DMZ.
5. Теплица ↔ дата-центр: MQTT/AMQP и видео — отдельные каналы; риск подделки устройства или перехвата на линии.

## Методология

Используется классификация **STRIDE** (Microsoft) для систематизации угроз:

| Категория | Суть |
|-----------|------|
| **S** Spoofing | Подмена субъекта или источника данных |
| **T** Tampering | Изменение данных или кода в пути или в покое |
| **R** Repudiation | Отрицание действий при отсутствии доказательной записи |
| **I** Information disclosure | Несанкционированное раскрытие данных |
| **D** Denial of service | Снижение доступности сервисов |
| **E** Elevation of privilege | Обход авторизации, рост привилегий |

Для каждой значимой зоны ниже перечислены типовые угрозы и **контрмеры на уровне архитектуры** (детальная реализация — в коде, конфигурации и политиках).

## Угрозы по зонам системы

### 1. Пользовательский веб-периметр (SPA, reverse proxy)

| ID | STRIDE | Угроза | Контрмеры (архитектура и практики) |
|----|--------|--------|-------------------------------------|
| T-PER-01 | S, I | Кража сессии / токена (XSS, утечка в localStorage, перехват без TLS) | CSP, минимизация inline-скриптов, короткий TTL access token, refresh с ротацией; только HTTPS; см. [ADR-0002](../../../architecture/adr/0002-react-typescript-frontend.md), поток OIDC с PKCE |
| T-PER-02 | T | Подмена ответа API (MITM) | TLS везде на периметре; HSTS при публичном DNS |
| T-PER-03 | D | Flood на GraphQL/WebSocket/API | Rate limiting на L7, лимиты размера запросов, таймауты; масштабирование stateless API ([NFR в calc_architecture](../../../architecture/calc_architecture.md)) |
| T-PER-04 | E | Обход авторизации на уровне API | Проверка scope/ролей на каждом защищённом endpoint; не полагаться только на скрытие UI |

### 2. Identity (OpenIddict, Identity DB)

| ID | STRIDE | Угроза | Контрмеры |
|----|--------|--------|-----------|
| T-ID-01 | S | Подмена клиента OIDC / redirect_uri | Зарегистрированные redirect URI, PKCE, проверка `state` |
| T-ID-02 | I | Утечка refresh-токенов из БД | Шифрование на диске для БД по политике; ограничение срока и отзыв; минимизация логирования токенов |
| T-ID-03 | E | Подбор паролей | Политика сложности, блокировки, при возможности MFA |
| T-ID-04 | R | Невозможность расследовать выдачу токенов | Аудит событий OpenIddict/приложения (успех/отказ входа), синхронизация времени NTP |

### 3. Web API и бизнес-логика

| ID | STRIDE | Угроза | Контрмеры |
|----|--------|--------|-----------|
| T-API-01 | E | IDOR / доступ к чужим объектам теплицы | Авторизация по **UUID** объекта и членству в организации из токена и `CNT_GM_DB` (не полагаться на неугадываемость UUID); тесты на горизонтальное повышение |
| T-API-02 | T, I | Инъекции в запросы к GraphQL/REST | Параметризация, allow-list полей, лимиты глубины GraphQL |

| T-IOT-02 | T | Подмена значений датчиков | Подпись полезной нагрузки (если введётся), контроль выбросов на стороне сервиса, алерты на аномалии |
