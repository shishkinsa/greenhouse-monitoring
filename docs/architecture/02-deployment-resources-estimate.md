# Оценка ресурсов для развёртывания (production)

Документ основан на:

- логической и deployment-модели: `docs/architecture/diagram/infrastructure/30-production-deployment.c4`;
- расчётах нагрузки и объёмов: `docs/architecture/01-calc-architecture.md`;
- масштабе и NFR: `docs/requirements/non-functional/04-scalability.md` (NFR-04).

**Назначение:** сводная таблица **vCPU**, **RAM** и **постоянного хранилища** (для подов Kubernetes — **PVC**; для виртуальных машин — **диск данных/системы**) для закупки инфраструктуры и планирования ёмкости worker-узлов и ВМ.

**Допущения:** значения — **проектные оценки** «на единицу» (под или ВМ); фактические лимиты/requests задаются в Helm/manifests и могут уточняться по нагрузочным тестам. Для MinIO объём PVC привязан к верхней оценке архива видео из `01-calc-architecture.md` (~162 ТБ полезных данных за 30 суток) и к режиму **distributed из 4 подов** с erasure coding (заложен запас на кодирование и служебные объёмы).

| Метрика | Значение |

## 3. Объём объектного хранилища (видео) и согласование с MinIO

По `01-calc-architecture.md` верхняя оценка архива видео (480p, 0,5 Мбит/с, 30 суток) — **до ~162 Ti** полезных данных. В таблице Kubernetes для MinIO заложено **~168 Ti** сырого PVC на распределённый кластер (4 пода по ~42 Ti) с запасом под **erasure coding** и небольшой недоукладкой по ровности дисков. При снижении битрейта, записи по событиям и lifecycle фактический объём будет ниже — пересчёт в ТП.

## 4. Периметр и плоскость управления (вне сводных таблиц 1–2)

Следующие элементы из модели развёртывания **не свёрнуты** в строки выше и задаются политикой платформы:

- **L7 / reverse proxy** (443), **TCP Ingress MQTT** (8883) — отдельные ВМ или сервис облака / DaemonSet; типично **0,5–2 vCPU** и **0,5–2 Gi RAM** на инстанс при HA.
- **Узлы Kubernetes (control plane + workers):** ёмкость worker’ов должна покрыть сумму из §1 плюс системные поды (CNI, DNS, metrics), **резерв ~15–25%** по CPU/RAM и диск под образы и логи.
- **Резервное копирование** БД и конфигураций — отдельное хранилище или политика snapshot (не включено в PVC приложений).

## 5. Сводный итог по вычислительным ресурсам и хранилищу

| Категория | vCPU | RAM | Постоянное хранилище (PVC или диск ВМ) |
|-----------|------|-----|----------------------------------------|
| **Kubernetes (приложения из §1)** | 27,25 | ~72 Gi | **~168,3 Ti** (в т.ч. MinIO; прочие PVC ≈260 Gi) |
| **Виртуальные машины (§2)** | 29 | 128 Gi | **~1,36 Ti** |
| **Всего (сложение)** | **56,25** | **~200 Gi** | **~169,7 Ti** (если MinIO и диски ВМ не дублируют друг друга; см. ниже) |

**Важно:** объём **~168 Ti** относится к **объектному хранилищу видео (MinIO PVC)**; диски **ВМ ClickHouse/PostgreSQL** — отдельный контур реляционной и аналитической СУБД (**~1,36 Ti**). Их **не суммируют** как один пул без учёта топологии: в закупке/облаке задаются **два класса хранилища** (блочные диски ВМ + объектное S3/MinIO).

### Итого для планирования закупки (кратко)

| Показатель | Значение |
|------------|----------|
| **Всего vCPU (приложения K8s + ВМ)** | **~56** (округление вверх для лицензий/квот) |
| **Всего RAM** | **~200 Gi** |
| **Хранилище: ВМ (блочное)** | **~1,4 Ti** |
| **Хранилище: объектное (MinIO / видеоархив)** | **~168 Ti** (верхняя оценка по calc) |
