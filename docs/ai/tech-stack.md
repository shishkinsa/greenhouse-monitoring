# Технологический стек (по архитектуре)

Свод основан на моделях LikeC4 в `docs/architecture/diagram/containers/` и `docs/architecture/diagram/context/`. При смене стека сначала обновляют архитектурные модели и требования, затем этот файл.

## Пояснение по выбору технологий

Ниже — сжатое обоснование в терминах архитектуры и расчётов нагрузки ([calc_architecture.md](../architecture/calc_architecture.md)); это не маркетинговое сравнение с альтернативами.

- **Единая серверная платформа (.NET / ASP.NET Core)** — один стек для основного API, сервиса сохранения телеметрии и Identity: проще сопровождение, общие практики безопасности и развёртывания. **OpenIddict** даёт стандартный OIDC/OAuth2 поверх того же ASP.NET Core, что согласуется с клиентской аутентификацией через **OpenID Connect**.
- **React и TypeScript** — типизированный UI для сложных экранов мониторинга и интеграции с API (GraphQL/WebSocket в модели связей).
- **RabbitMQ с MQTT** — контроллеры публикуют телеметрию по **MQTT** (типичный выбор для IoT); брокер с MQTT-плагином объединяет поток устройств и подписку внутренних сервисов по **AMQP** без дублирования контура (см. расчёт постоянных MQTT-соединений в `calc_architecture.md`).
- **PostgreSQL и ClickHouse** — разделение ролей: реляционная модель для справочников, метаданных и Identity; колоночное хранилище для больших объёмов временных рядов и поиска событий. **Redis** — низкая задержка для последних показаний и сессий без перегрузки СУБД частыми мелкими чтениями.
- **MinIO (S3 API)** — объектное хранилище для крупных неизменяемых сегментов архива видео; **go2rtc** — специализированный слой RTSP/WebRTC между камерами и браузером.
- **HashiCorp Vault** — централизованная выдача и ротация секретов для нескольких сервисов вместо разбросанных конфигов.
- **TLS на периметре и между компонентами** (как в подписях связей в моделях), **reverse proxy / балансировщик** — завершение TLS и горизонтальное масштабирование stateless API при росте нагрузки (**NFR-03** в расчётах архитектуры).

## Клиентские приложения

| Технология | Где используется |
| --- | --- |
| React 18.2.0, TypeScript | `CNT_GM_Web`, `CNT_GM_Identity_Web` (метаданные в модели Web) |

## Серверные приложения и API

| Технология | Где используется |
| --- | --- |
| ASP.NET Core, .NET 10 | `CNT_GM_WebAPI`, `CNT_GM_SavingService`, `CNT_GM_Identity_WebAPI` |
| OpenIddict | `CNT_GM_Identity_WebAPI` (поверх ASP.NET Core) |

## Протоколы и API между компонентами

| Технология | Назначение |
| --- | --- |
| HTTPS, WebSocket, GraphQL | Клиент ↔ Web API; Bearer access token |
| OpenID Connect | Аутентификация Web-приложения с Identity |
| HTTPS, OIDC metadata, JWKS | Web API ↔ Identity Web API (проверка access token) |
| TLS/MQTT | Контроллеры теплицы → брокер |
| AMQP (5672/TCP via TLS) | `CNT_GM_SavingService` → брокер |
| HTTPS, Vault API | Сервисы → секреты |

## Обмен сообщениями

| Технология | Назначение |
| --- | --- |
| RabbitMQ с MQTT-плагином | Внешний MQTT для устройств; внутренние подписчики по AMQP |

## Данные и кэш

| Технология | Назначение |
| --- | --- |
| PostgreSQL | Метаданные и справочники (`CNT_GM_DB`); Identity (`CNT_GM_Identity_DB`) |
| ClickHouse | Телеметрия и аналитика датчиков (`CNT_GM_Timeseries_DB`); порты приложений: 8123 (запись), 8443 (доступ API из Web API по модели связей) |
| Redis | Кэш последних показаний, сессии (`CNT_GM_Redis_DB`) |

## Видео

| Технология | Назначение |
| --- | --- |
| go2rtc | Живое и архивное видео (`CNT_GM_WebRTCServer_Only`, `CNT_GM_WebRTCServer_History`) |
| RTSP (554/TCP), UDP RTC (5004, 5005) | Камеры → go2rtc |
| WebSocket, MSE | Браузер ↔ go2rtc (по модели Web) |
| MinIO (S3 API) | Архив сегментов видео (`CNT_GM_S3`) |

## Инфраструктура и секреты

| Технология | Назначение |
| --- | --- |
| HashiCorp Vault | Хранение секретов (`CNT_GM_Secrets`) |
| Reverse proxy / L7 load balancer | TLS на периметре, балансировка Web и API (описано в модели Web и Web API) |

| Технология | Назначение |
| --- | --- |
| MQTT (TLS) | Публикация телеметрии с цифрового контроллера в брокер |
